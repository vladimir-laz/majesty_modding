# Рецепт: спавн логов и стартовые деньги — в файлах МИССИЙ (.mi)

## Ключевой факт
Кампания **переопределяет спавн и деньги per-mission** в файле миссии
`resource/map/.../<mission>.mi`. Глобальный
[[global-spawn-settings|global_spawn_settings.set]] кампанийные лога НЕ
используют (он работает для логов, построенных ИГРОКОМ, и для немногих
миссий без своей коллекции). Поэтому правка глобального файла не меняет число
монстров в логах кампании — нужно править `.mi`. Файлов миссий ~90, ~54 МБ →
держать в моде накладно (мы их кладём только в локальную сборку, не в git).

## 1) Спавн монстров из логов — `helper/spawn_program_collection` в .mi
```
{"helper/spawn_program_collection"
  {"<имя>"
    {raw <монстр>}
    {"AI task" {task protect}{target_tag <логово>}}  ; гарнизон логова
    {count "1 1"} {"spawn time" 45} {"max count" 3}
  }
  ...
}
```
**Типы `task` (важно):**
- `protect` / `none` — **стационарный гарнизон логова** (это «монстры в логове»). Увеличиваем.
- `attack_in_zone` — **волны атаки** на игрока (часто count/max 20–30). НЕ трогать.
- `waypoints` — патрули/роуминг. Обычно не трогаем.

Рычаги: `count "min max"` (за спавн), `"max count"` (одновременно у логова),
`"spawn time"` (период).

**Флаг `{"First wave"}` — про ранний перебор.** Программа с этим флагом
вываливает пачку **сразу на старте миссии**. Если поднять её `count` —
получится «лавина монстров в день 1». Решение: у first-wave программ `count`
оставлять малым (мягкий старт), а `max count` поднимать (лог дорастёт
постепенно). Из ~991 логовой программы ~369 — с First wave.

Пример (сборка more_monsters_and_start_money, усилено 2026-06, пересчёт от
БАЗОВЫХ миссий — идемпотентно): `none`+`protect` → `max count` **×4**,
`spawn time` **÷3** (пол 8), у НЕ-first-wave `count` = `max(3, base_max×2)`;
у first-wave `count` оставлен базовым (мягкий старт), но потолок/кулдаун тоже
усилены. `attack_in_zone`/`waypoints` не трогали. Итог: 991 программа в 86 миссиях.
> Пересчёт от базы (`resource/map/...`), а не от текущего билда — иначе значения
> компаундятся при повторном прогоне. Правятся только 3 поля спавна, прочие
> правки миссии (деньги, требования кузницы) сохраняются.

## 2) Стартовые деньги — `{"kingdoms"}` в .mi
```
{kingdom {kingdomdescriptor {kingdomdescriptor "player"} {humanplayer 1}}
    {kingdomtreasure "money" 2000 "wood" 0 "id" 1}   ; деньги ИГРОКА
}
```
Деньги игрока = `kingdomtreasure "money" N` в королевстве с **`{humanplayer 1}`**
(не по `id` — он разный в разных миссиях). Враги — `humanplayer 0`, не трогать.
Пример: всем игрокам поставили `max(20000, текущее)`.

## 3) Что ещё контролирует миссия (не только спавн/деньги)
**Доступность построек.** Здание может быть в `shops.set` мода (напр.
элитные башни `unique_tower_*`), но в конкретной миссии его не построить —
миссия/кампания задаёт свой список доступного. Поэтому «нет великих башен в
миссии» — это настройка миссии, а не отсутствие в моде.

## 4) ⭐ Требования к АПГРЕЙДУ зданий — тоже per-mission!
**Ключ к загадке «кузница монстров требует замок 2, хотя в shops.set убрано».**
Каждая миссия дублирует список зданий с их требованиями в блоках `{commodity "X"}`
(внутри `{shop "<замок>"}`). Там у каждого здания свои `upgrade_prerequsity`:
```
{commodity "mks_smithy"
    ...
    {upgrade_prerequsity 0 {building_prerequsity {buildings "mks_castle_2"}}}  ; апгрейд до ур.2
    {upgrade_prerequsity 1 {building_prerequsity {buildings "mks_castle_3"}}}  ; апгрейд до ур.3
}
```
Это **перекрывает** `shops.set` в кампании. Чтобы апгрейд не требовал замка —
обнулить: `{buildings ""}` (пустая строка = нет требования; такой вид уже
встречается в оригинале). Делать брейс-волком по блоку нужного `{commodity}`,
менять только `{building_prerequsity {buildings "..."}` внутри него.
Сделано для `smithy` и `mks_smithy` в сборке more_monsters: 81 миссия, 324 снятия.
> ⚠️ Если добавляешь зданию НОВЫЙ уровень апгрейда (напр. кузница ур.4), нужно
> добавить ещё `{upgrade_prerequsity 2 ...}` в commodity КАЖДОЙ миссии — иначе
> в кампании 4-й уровень будет недоступен.
